Diario del progetto:
06/06/2026 - Today we decided the task of each member, and started deciding the design of the project.

Implemented the user authentication (login) feature as a complete end-to-end RPC slice, covering the "Access & Registration" part of Identity & Profile Management. On the shared module we defined the User data object and the AuthService/AuthServiceAsync interfaces; on the server we added AuthServiceImpl with temporary in-memory user storage and a seeded test account, registered as authServlet (/app/auth) in web.xml; and on the client we built the LoginGui screen and updated App.java to load it as the entry point. During testing the server returned HTTP 503 because the servlet failed to deploy on Jetty 12/Jakarta: the cause was extending the legacy javax-based RemoteServiceServlet, which we fixed by switching to the Jakarta variant (com.google.gwt.user.server.rpc.jakarta.RemoteServiceServlet), establishing the convention that all server-side *Impl classes must use it. Finally, we standardized the new code to English across identifiers, comments, and user-facing messages.

07/06/2026 - 
Implemented the user registration screen (RegisterGui) on the client, completing the authentication flow started with the login feature. The new screen collects username, email, and password, validates that no field is empty, and calls the existing register method on AuthService via RPC; on success it notifies the user and redirects to the login screen, and on failure it displays the server-side error message (e.g. duplicate email or password too short). We also enabled the navigation link in LoginGui so the "Sign up" button now opens RegisterGui, giving a complete two-way flow between login and registration. This closes the "Access & Registration" part of Identity & Profile Management at the UI level; user data is still stored in memory on the server, with database persistence (MapDB) planned as a later step.

Implemented the post-login home screen (HomeGui) on the client, establishing the navigation pattern for the authenticated area of the app. The screen takes the logged-in User through its constructor, displays a personalized welcome with the user's username and email, and provides a sign-out button that returns to the login screen. We replaced the temporary placeholder alert in LoginGui with real navigation, so a successful login now redirects to HomeGui and passes along the authenticated user. This completes the full authentication cycle at the UI level (Login → Home → Sign out, and Register → Login → Home) and sets up HomeGui as the entry point from which the main features (Skill Marketplace, Profile, etc.) will later be reachable.

Implemented the announcement backend foundation for the SkillShare marketplace. In the shared module, we added the serializable `Announcement` data object with the announcement ID, owner username, offered skill, requested skill, description, availability, and active status. On the server, we created `AnnouncementRepository` to persist announcements in MapDB and `AnnouncementManager` to keep validation and business rules separate from persistence. The implementation supports announcement creation, retrieval by ID, active announcement listing, deactivation, and case-insensitive search across offered skill, requested skill, and description. Duplicate IDs are rejected atomically through `putIfAbsent`, preventing an existing announcement from being overwritten. We also added isolated in-memory unit tests for repository and manager behavior.

Extended the announcement functionality with a complete GWT RPC layer. In the shared module, we added `AnnouncementService` and `AnnouncementServiceAsync`; on the server, we created `AnnouncementServiceImpl` using the Jakarta-compatible `RemoteServiceServlet` and registered the new `announcementsServlet` endpoint (`/app/announcements`) in `web.xml`. The servlet delegates all operations to `AnnouncementManager`, avoiding duplicated validation logic and direct MapDB access. We also added RPC service tests covering creation, duplicate rejection, retrieval, active listing, deactivation, case-insensitive search, and validation errors. The announcement backend and RPC test suite currently passes successfully with 24 tests and no failures.

Migrated user persistence from in-memory storage to MapDB, so registered users now survive server restarts. We replaced the temporary HashMap in AuthServiceImpl with two persistent MapDB collections (email → password and email → username), opened through the template's DatabaseCore singleton via getDB(), and we call DatabaseCore.commit() after each registration to flush the changes to disk (progetto_sweng.db). Passwords are stored server-side only and are never included in the User object sent to the client. A test account is seeded on first run if it doesn't already exist. We verified persistence end-to-end: a user registered before a clean server restart could still sign in afterwards. This satisfies the project's data-persistence requirement and establishes the MapDB usage pattern (open collection → modify → commit) that the remaining services will reuse.
